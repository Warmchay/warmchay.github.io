#5 LLVM IR

IR: Intermediate representation (中间表示)

5.1 IR

编译器工作过程可以分为三个部分：前端(front-end) - 中端(middle-end) - 后端(back-end)，每个部分的工作为：

  • 前端：源语言 –> IR ( Clang、Rustc 等)
  • 中端：优化 IR （ LLVM Optimizer ）
  • 后端：IR 翻译为目标语言，一般是目标体系结构的汇编，如MIPS/x86 （ LLVM static compiler ）
Fig 1: LLVM 工作流程

这种分工做法的好处是：

  1. 只需要在 IR 上做好优化，再翻译到汇编，这样能够在所有支持的体系结构上实现优化，减少了重复的工作量
  2. 举个例子，有 m 种语言，n 类编译器，如果直接将源语言翻译成目标语言，需要写 m*n 个编译器。而如果使用 IR 的中间形式 ，只需要 m+n 个编译器。

LLVM IR 三种等价的中间格式为：

  • 在内存中的编译中间语言 (无法通过文件的形式展示)
  • 硬盘上存储的二进制中间语言 (.bc)
  • 人可读的代码语言 (.ll)

1 如何在 Linux 中修改目录权限

Chmod = short for change mode

  • chmod +rwx filename: add permissions
  • chmod -rwx filename: remove permissions
  • chmod +x filename: 允许执行(x = execute)
  • chmod -wx filename: 不允许写/执行

举个例子：

  1. 第一个符号。- 表示文件，d表示文件夹dir，l表示链接link
  2. 第一个rwx的位置，提供rwx权限给本文件的owner
  3. 第二个rwx的位置，提供rwx权限给有这个文件的一组用户
  4. 第三个rwx的位置，提供rwx权限给其他用户
  5. 第一个 root 表示该文件的 user ， 第二个 root 表示该文件归属于哪一个 group

2 修改 Group Owners/ Others 的文件权限

  • chmod g+w filename (group + write)

  • chmod g-wx filename (group - wx)

  • chmod o+w filename (others + write)

  • chmod ugo+rwx foldername (users&&group&&others + rwx) == (everyone + rwx)

  • chmod a=r foldername (所有人都只能 read)

3 修改文件的 Owner/Group

Change group (Chgrp)

  • chgrp groupname filename/foldername

Change ownership (Chown)

  • chown name filename/foldername

Change group&&owner

  • chown -R name:group /home/name/dirname

4 用数字修改文件权限

二进制 $111_{(7)}$ 分别表示 $rwx$ 的位置，所以为 0 表示没任何权限，1 表示可执行，2 表示可写，4 表示可读，7 表示可读写执行。

  • chmod 777 foldername: 给所有人 rwx 的权限
  • chmod 700 foldername: 只有 user 能 rwx
  • chmod 327 foldername: owner 可以 -wx， group 可以 w， 其他人可以rwx

Summary

解答疑问：

  1. 2进制方便读写，rwx格式给人读的
  2. chmod/chown/chgrp 等都是 change + 对象 的缩写
  3. 第一个字符表示的是操作的是 文件/文件夹/链接

（1）Http特点

  1. 支持C/S架构：也是一种请求/响应模式的协议
  2. 简单快速： 客户向服务器请求服务时，只需要传送「请求方法」和路径即可，请求方法常用的是GET、HEAD、POST；

Https: HyperText Transfer Protocol over Secure Socket Layer

为：Http+SSL/TLS，通过SSL证书来验证服务器身份，为浏览器和服务器之间的通信进行加密

SSL（Secure Socket Layer）： 安全套接字层，在TCP/IP协议和各种应用层协议之间，为数据通讯提供安全支持

TLS（Transport Layer Security): 传输层安全性协议，采用非对称加密对通信方做身份认证，目前Google、FaceBook在以这个协议来创建安全连线。

Https 数据传输流程：

KeyPoints：

  • Client 通过 URL 访问 Server 建立 SSL 连接
  • 客户端将网站的公钥加密为会话密钥，后传送给网站
  • 服务器通过会话密钥加密其与Client之间的通信

缺陷：

  1. 多次握手，导致页面加载时间延长了近50%
  2. 连接缓存没有Http高效，增加了数据开销和功耗
  3. 申请SSL需要money，功能越强大的证书费用会越高
  4. SSL涉及到的安全算法会消耗CPU资源，消耗服务器资源较大

Https 与 Http 的区别

  1. Http协议的数据传输为明文，不安全；Https用SSL/TLS协议对内容进行了加密处理
  2. Http和Https使用连接的方式也不同，Http默认端口为80，Https默认端口为443

A.2 BitTorrent

1. 相关概念

  • Tracker： 收集下载者信息的服务器，并把这个信息提供给其他的下载者，让下载者们集合起来一起传输数据；
  • 种子： 如果一个下载任务中的所有文件都被下载者A完整的download了，那么此时A成为一个种子，发布者本身发布的文件就是原始种子；
  • 做种
    1. 发布者提供了下载任务的全部内容
    2. 下载者下载完成后继续提供给其他人下载

2. 过程

文件发布者根据要发布的文件生成一个.torrent文件，即“种子”；

种子包含了Tracker信息和文件的信息，主要原理为把下载的文件虚拟分成大小相等的块，并且把每个块的索引信息和Hash验证码写入种子中，此时种子文件即为下载文件的“index”。

下载者要下载文件内容，需要先得到相应的种子文件，再使用BT客户端软件下载

BT客户端先解析种子文件得到Tracker地址，后连接到Tracker服务器，Tracker服务器回应下载者请求，提供下载者和其他下载者的IP，下载者再连接其他下载者，根据种子文件，双方交换彼此没有的数据。

因为这个过程不需要其他服务器参与，分散了单个线路上的数据流量，减轻了服务器的负担，BT下载的特点是：下载的人越多，提供的带宽也越多，下载速度就会越快。

What is FPGA

FPGA: Field Programmable Gate Array(门阵列)

An FPGA is an integrated circuit consisting of “logic blocks”, “I/O cells” and “interconnection resources”,this allows the chip to be reconfigured to connect the I/O and logic blocks together.

1. logic blocks

each could do simple logic ops, such as AND or XOR, and has some degree of memory(存储器), be it a simple Flip-flop（触发器） or a more complex block of memory.

And logic blocks have turned to by more logic function blocks using lookup tables(查找表) within the blocks to switch current function: e.g. to perform tasks as arithmetic ops.

Current FPGA are starting to cross the line of FPGA and on-chip system, while the true power of FPGA is having a nember of logic blocks that can be connected together to perform massively parallel, and real time processing..

Speed Comparison

  • CPU: how long functions take to exec or how fast they run
  • FPGA: based on data throughput.

Sweet Point:

in-line processing（内联处理），可以在不知道其余数据是什么样的情况下对data stream执行功能，利于管理高数据速率，仅需要 CPU 去处理少量数据。

Weak Point：

内存小，没有寻址寄存器组，无法处理数据。

1. GPU 瓶颈之一在于运算单元喂不饱

GPU有很多并行运算单元，只有当运算单元一直在做运算时才能发挥最大效率。当代码写的不好，会出现GPU没做多少计算就需要从DDR取数据，运算单元则有可能因为取数据的延时而在那里等待，这样GPU运算单元的利用率就低了。

所以GPU要提高效率，需要充分利用片上内存，减少DDR的访问频率，这和CPU要优化Cache命中率是一样的。

2. FPGA的优势在哪

如Tesla P40与其附带的FPGA的原始算力是差不多的，但是在片上内存上，FPGA有着更高的计算容量，片上内存在DL中多应用于减少掩饰，大量的片上Cache缓存能减少外部内存读取带来的内存瓶颈，能减少高内存带宽解决方案所需要的功耗和成本。

访问外部存储，如读写DDR，是非常耗能的，可能数据在DDR和芯片之间的能耗比芯片本身做计算的能耗还要高，FPGA片上内存的大容量和灵活配置能减少对外部DDR的读写，这个可以缓解内存瓶颈，也能减少延时和降低功耗。

3. FPGA最擅长实时流水线计算

实时的意思即运算速度和数据来的速度是一样的，如1920*1080*60帧的视频流，像素时钟频率为150MHz，FPGA在对它进行处理的时候运算时钟频率也为150MHz。

FPGA能在图像一边从HDMI或传感器芯片传过来时同时做处理，而CPU则是one by one，因此FPGA可以把大量计算全部并行起来达到实时速度的。

4. 最高效的计算方式应该是

计算的瓶颈往往不在于运算单元的数量，而是在内存带宽上，即数据读写来不及，运算单元喂不饱，最高效的计算方式是能利用片上的高速内存，这样能尽可能减少对外部DDR的读写，这也同时降低了功耗。

为什么FPGA比GPU的能耗更低呢？因为FPGA是硬件可编程，所以它有着直接的数据通路。比如FPGA算两个数相加，那直接把两个数从内存读进来送到加法器后去加即可，但是在GPU中还需要进行指令译码，译码结果为加法后才能送往加法器，这中间还需要经过一些选通器才会被正确的送到加法器而不是其他的运算单元。

而GPU这些译码，选通数据都会产生额外的功耗，但是FPGA每次需要做的运算都是固定的，不需要指令译码，也不需要数据的选通。

FPGA相较于GPU最大的优势为其有片上内存，比GPU的Cache大得多，可以缓存需要运算的数据和中间结果，减少从DDR中读数据

1. SDRAM，DRAM，SRAM and RAM、ROM

  • ROM and RAM

ROM：read-only，储存资料后不能改变，资料不会因为关闭电源而消失；

RAM： random-access，自由指定地址进行数据读写，但是掉电之后就会丢失数据：如PC的内存。

  • SRAM and DRAM

SRAM： Static RAM，静止存取功能的内存，不需要refresh电路即能保存它内部存储的数据，在加电情况下不需要refresh，数据不会丢失。但集成度较低，相同容量的内存需要很大的体积，功耗较大，同时因为比较昂贵，常用在CPU的一级、二级Cache

DRAM： Dynamic RAM，只能将数据保存较短时间，为了保持数据，DRAM使用电容储存，隔一段时间需要refresh一次，如果存储单元没有被refresh，存储的信息就会丢失。速度比SRAM慢，但是价格便宜，常用于PC的内存。

Synchronous DRAM：同步动态随机存储器，Synchronous指Mem工作需要同步时钟，内部命令的发送与数据的传输都以它为基准。

DDR： 在SDRAM基础上发展而来，这种改进型的DRAM和SDRAM基本一样，不同之处在于可以它可以在一个时钟读写两次数据，这样使得数据传输速度加倍了。

2.1 Alioth内核

2.1.1 概述

​ Alioth 微内核主要针对专业的应用场景，来提高系统的扩展性，使系统在经历不可避免的变更时拥有较强的灵活性，并且提供针对提供这样的灵活性所需要付出的成本间的平衡能力。当往系统中添加新业务时，不需要改变原有组件，只需把新业务封闭在一个新的组件中就能完成整体业务升级。

2.1.2 背景

​ 内核作为直接运行在硬件上的最基础的软件实体，是操作系统的核心部分，管理着系统的各种资源。目前从内核架构来划分，可以分为 微内核（Micro Kernel）和 宏内核（Monolithic Kernel），常见的如 Linux、Windows 等系统都为宏内核，它们的用户服务和内核服务运行在相同的地址空间中，是一个包含了非常多功能的底层程序。

​ 宏内核的功能完备，但是是不可插拔的，修改一些小的功能，都会涉及到整个程序的重新编译等；并且如果其单个服务出现 bug导致崩溃，往往意味着整个系统的崩溃，可靠性不强。

2.1.3 功能描述

1 Plug-in Architecture

微内核架构包含两类组件：核心系统和插件模块，核心系统负责和具体业务功能无关的通用功能，如模块加载、IPC等；插件模块负责实现具体的业务逻辑。

核心系统功能比较稳定，不会因为业务功能扩展而不断修改，插件根据业务功能的需要不断地扩展。

重点特性：

  • 插件注册管理

核心系统提供插件注册表（可以是配置文件、代码或者数据库等），插件注册表含有每个插件模块的信息，包括它的名称、数据协议、远程访问协议等，核心系统通过注册表知道哪些插件模块是可加载的并且如何使用它们。

  • 插件连接

核心系统依据 OSGI规范 连接插件，之后插件按照规范实现，核心系统按照规范加载。

  • 插件通讯

插件间虽然是完全解耦的，但实际业务运行中，必然出现某个业务流程需要多个插件协作，这就要求两个插件间进行通信。采用 HTTP、HSF 等通讯协议，由内核告知通讯双方各自的地址。核心系统通过创建基于消息的进程间通讯机制，提供 Send 和 Receive 接口实现插件间异步通信，

  • 总线设计机制

使用 EventBus 结构，使得应用内部在做 Plug-in 组件解耦时，进程能通过总线结构向内核发送消息，之后再由内核发送给对应的进程。

2 Broker Bus

核心系统作为插件之间消息通讯的路由，由 Broker 集中进行插件通讯，管理插件的服务、插件动态更新等，只添加必须的功能。编写插件，使插件通过与 Broker 之间进行收发消息，来扩展系统更多的功能。

如果插件实例的负载太高，插件会从Broker上采集数据，分析后调用K8s API进行动态扩容。并且系统将通过使用私有云端平台的KV存储，及时将用户数据加密上云。

2.2 Vec

2.2.1 概述

Vec为基于Alioth内核补丁的资源虚拟层，它启动的多个域（Domain）能同时存在一个机器上，在Alioth中Vec管理着Alioth微内核域和RTOS简易实时内核域。

所有的域根据获得的全系统的优先级，有序地处理外部事件（如中断）或内部事件（如陷阱、异常）。

2.2.2 背景

在Alioth内核中，其调度算法（基于最大吞吐量准则）、设备驱动、不可中断的系统调用、中断屏蔽以及虚拟内存的使用等因素，会导致系统在时间上的不可预测性，决定了Alioth无法处理硬实时任务。

而机器人会经常遇到需要迅速作出决策的场景，如果不能及时反应，机器人会出现宕机甚至完全损坏的情况，为了确保高优先级任务能在第一时间内全部完成，Vec在Alioth微内核与硬件间加一个虚拟层，同时创建一个简易的、时间上可以预测的实时内核，来保证运行的实时进程能满足硬实时性。

在Vec的调控下，Alioth能完成既包括实时部分又包括非实时部分的复杂任务。

2.2.3 功能描述

Vec 作为Alioth下的软件层，通过向上层多个内核提供某些原语和机制来实现硬件共享，而Vec并不会对硬件的使用强加任何的限制，上层的内核仍然可以自由的操作硬件，不会因为Vec的存在而有任何的约束。

1 Domain

每个内核都能在独立的域（Domain）内运行，每个域都有独立的地址空间和类似于进程、虚拟内存的软件抽象层，并且这些资源也能由不同的域共享。

关于域与硬件资源的交互操作分为四类：

  • A类交互：域对硬件的直接操作，此类情况下，忽视Vec的存在
  • B类交互：一方面Vec接受硬件产生的中断和异常，另一方面，Vec也能直接控制硬件；
  • C类交互：Vec接受到硬件中断后，执行相应域的中断服务程序；
  • D类交互：当域内的内核知道有Vec存在时，它可以主动向Vec请求某些服务，如共享其他域中的资源、请求授权域优先级等，实现各个域之间的通讯。
2 I-Pipe

Vec通过I-Pipe（Interrupt Pipe）在不同域之间传播中断，提供相应的机制让域改变自己在I-Pipe中的优先级，并通过控制系统的中断来实现对各个域内内核的控制。

I-Pipe对域中断的处理方式有四种：

  1. 允许中断域

中断产生后，Vec会调用相应域的中断处理程序，这和不存在Vec的情况是类似的，只不过在这种情况下，中断服务程序由Vec负责调用。

  1. 禁止中断域

每个域设置软件标志，标定是否禁止中断，如果某个域禁止中断，则硬件中断将沿着中断管道进一步向下传播。

  1. 抛弃中断域

当中断传播到这类域时，Vec不做任何的处理，直接将这个中断沿着I-Pipe向后传播。

  1. 终止中断域

中断传播到这类域时，Vec根据这类域的设置处理完传播的中断后，不再沿着I-Pipe向后传播，后面低优先级的域不会知道有这个硬件中断的产生。

3 Event Pipeline

Vec基本结构是客户域征求事件控制链，域作为基于内核的软件组件能通过Event Pipeline通知Vec：传入的外部中断或自动产生的虚拟终端、Alioth应用程序发出的每个系统调用、内核代码触发的系统事件（如任务切换、信号通知、任务退出等）

Vec为每个域分配静态优先级严格定义事件到域的传递顺序，所有活动域根据其优先级排队，形成从最高优先级域到最低优先级域的管道抽象。之后传入的事件（包括中断）将被推送到管道的头部（最高优先域处）向下进行到尾部。

Alioth微内核作为根域，RTOS域需要通过Alioth来加载它的内核模块。

4 OIP

Vec通过OIP（Optimistic interrupt protection）以优先级的方式分发中断，同时允许域创建无中断部分，能够推迟给定域占用管道的阶段。

  • 当一个阶段被推迟，暂停的中断将累积在该域的中断日志（i-log）中，当该阶段不再被推迟时，根据日志记录进行同步处理；

  • 当一个域处理完它所接收到的待处理的中断后，它会调用一个特殊的Vec服务让出CPU到管道的下一个域，因此后者可以依次处理唤醒它的待处理的事件，并且这个周期会持续到管道的最低优先级域。

通过OIP的优先处理方式，高优先级域能接收中断并最终可抢占任何低优先级域，保证了提前运行的实时系统在管道的任何时候能够接收中断，不产生任何延迟。

2.2.4 应用场景

而在大多数行业中，数据是以孤岛的形式存在的，由于行业竞争、隐私安全、行政手段复杂等问题，即使在同一公司的不同部门之间实现数据整合也面临着重重阻力，在现实中想要将分散在各地、各个机构的数据进行整合几乎是不可能的，所需的成本巨大。

样本对齐

盲签名算法RSA

HexFed允许消息者先将信息盲化，后让签名者对盲化的消息进行签名，最后消息拥有者对签字除去盲因子，得到签名者关于原消息的签名。

盲签名为接收者在不让签名者获取所签署消息具体内容的情况下所采取的一种特殊数字签名技术，除了满足一般的数字签名条件外，还需要满足下面两条性质：

  1. 签名者对其所签署的消息是不可见的，即签名者不知道他所签署消息的具体内容；
  2. 签名消息不可追踪，即当签名消息被公布时，签名者无法知道这次是他哪次签署的。

HexFed通过多次哈希方程对原始结果进行加密，并采用盲签名来确保消息是被准确传递的，保证了计算结果不存在假阳性的可能。

1.1进程概览

进程让一个程序可以假设它独占一台机器，进程让程序提供“看上去”私有的，其他进程无法读写的内存系统（或地址空间），以及一颗“看上去”仅执行该程序的CPU

xv6使用页表（由硬件实现）来为每个进程提供其独有的地址空间，页表将虚拟地址映射为物理地址（cpu向主存发送的地址）

如图所示，一片地址空间包含了从虚拟地址0开始的用户内存，它的地址最低处放置进程的指令，接下来则是全局变量，栈区，以及一个用户可按需拓展的“堆”区（malloc用）

和上面提到的用户内存一样，内核的指令和数据也会被进程映射到每个进程的地址空间中，当进程使用系统调用时，系统调用实际上会在进程地址空间中的内核区域执行。

这种设计使得内核的系统调用代码可以直接指向用户内存，为了给用户留下足够的内存空间，xv6将内核映射到了地址空间的高地址处，即从0x80100000开始

xv6使用结构体struct proc来维护一个进程的状态，其中最为重要的状态是进程的页表，内核栈，当前运行状态。我们接下来会用p->xxx来指代proc结构中的元素。

每个进程都有一个运行线程，来执行进程的指令。线程可以被暂时挂起，稍后再恢复运行，系统在进程之间切换实际上就是挂起当前运行的线程，恢复另一个进程的线程，线程的大多数状态（局部变量 & 函数调用的返回地址）都保存在线程的栈上。

每个进程都有用户栈和内存栈（p->kstack），当进程运行用户指令时，只有其用户栈被使用，其内核栈则是空的，然而当进程（通过系统调用或中断）进入内核时，内核代码就在进程的内核栈中执行。

进程处于内核中时，其用户栈仍然保存着数据，只是暂时处于不活跃状态，进程的线程交替地使用着用户栈和内核栈。

需要注意内核栈是用户代码无法使用的，这样即使一个进程破坏了自己的用户栈，内核也能保持运行。

  • 当进程使用系统调用时，处理器转入内核栈中，提升硬件的特权级，然后运行系统调用对应的内核代码。

  • 当系统调用完成时，又从内核空间回到用户空间，降低硬件特权级，转入用户栈，恢复执行系统调用指令后面的用户指令。线程可以在内核中“阻塞”等待I/O，在I/O结束后再恢复运行

p->state表示了进程的状态：新建、准备运行、运行、等待I/O或退出状态中

p->pgdir以x86硬件要求的格式保存了进程的页表，xv6让分页硬件在进程运行时使用p->pgdir，进程的页表还记录了保存进程内存的物理页地址

1.3 代码：创建第一个进程

在main初始化了一些设备和子系统后，它通过调用userinit(1239)建立了第一个进程。userinit首先调用allocproc，allocproc（2205）的工作是在页表中分配一个槽（即结构体struct proc）并初始化进程的状态，为其内核线程的运行做准备。

注意： userinit仅仅在创建第一个进程时被调用，而allocproc创建每个进程时都会被调用，allocproc会在proc的表中找到一个标记为UNUSED(2211-2213)的槽位。

当它找到这样一个未使用的槽位后，allocproc将其状态设置为EMBRYO，使其被标记为被使用的并给这个进程一个独有的pid（2201-2219）。

接下来，它尝试为进程的内核线程分配内核栈，如果分配失败，allocproc会把这个槽位状态恢复为UNUSED并返回0来标记失败。

现在allocproc必须设置新进程的内核栈，allocproc以巧妙的方式，使其既能在创建第一个进程时被使用，又能在fork操作时被使用。

allocproc为新进程设置好一个特别准备的内核栈和一系列内核寄存器，使得进程第一次运行时会“返回”到用户空间。

1.4 运行第一个进程

现在第一个进程的状态已经被设置好了，开始运行它。

在main调用了userinit后，mpmain调用schduler开始运行进程（1267），scheduler(2458)会找到一个p->state = RUNNABLE的进程initproc，然后将per-cpu的变量proc改为该进程，接着调用switchuvm通知硬件开始使用目标进程的页表（1768）。

因为setupkvm使得所有进程的页表都有一份相同的映射，指向内核的代码和数据，所以当内核运行时我们改变页表是没有问题的，switchuvm同时还设置好任务状态段SEG_TSS，让硬件在进程的内核栈中执行系统调用与中断

scheduler接着把进程的p->state = RUNNING，调用swtch(2708)，切换上下文到目标进程到内核线程中。swtch会保存当前的寄存器，并把目标内核线程中保存的寄存器（proc->context）载入到x86的硬件寄存器中，其中也包括栈指针和指令指针。

当前的上下文并非是进程的，而是一个特殊的per-cpu调度器的上下文，所以scheduler会让swtch把当前的硬件寄存器保存在per-cpu的存储（cpu->scheduler）中，而非进程的内核线程上下文中。

最后的ret（2727）指令从栈中弹出目标进程的%eip，从而结束上下文切换工作，现在处理器就运行在进程p的内核栈上了。

allocproc通过把initproc的p->context->eip设置为forkret使得ret开始执行forkret的代码；第一次被使用时，forkret(2533)会调用一些初始化函数。

注意：我们不能在main中调用他们，因为它们必须在一个拥有自己的内核栈的普通进程中运行

接下来forkret返回，由于allocproc的设计，目前栈上在p->context之后即将被弹出的是trapret。因而接下来会运行trapret，此时%esp保存着p->tf，trapret(3027)用弹出指令从trap frame(0602)中恢复寄存器，就像swtch对内核上下文的操作一样：popal恢复通用寄存器，popl恢复%gs, %fs, %es, %ds.addl跳过trapno和errnode两个数据，最后iret弹出%cs, %eip, %flags, %esp, %ss。trap frame的内容已经转移到CPU状态中，所以处理器会从trap frame中%eip的值继续执行，对于initproc来说，这个值就是虚拟地址0，即initcoude.S的第一个指令。

这时%eip和%esp的值为0和4096，这时进程地址空间中的虚拟地址，处理器的分页硬件会把它们翻译为物理地址，allocuvm为进程建立了页表，所以现在虚拟地址0会指向为该进程分配的物理地址处。

allocuvm还会设置PTE_U来让分页硬件允许用户代码访问内存，userinit设置了%cs的低位，使得进程的用户代码运行在CPL = 3的情况下，这意味着用户代码只能使用带有PTE_U设置的页，并且无法修改像%cr3这样的敏感硬件寄存器，这样，处理器就受限只能使用自己的内存了。

1.5 第一个系统调用

initcode.S做的第一件事为出触发exec系统调用，如在第0章看到的一样，exec用一个新的程序来代替当前进程的内存和寄存器，但是其文件描述符、进程id和父进程都是不变的。

initcode.S(7708)刚开始会将$argv, $init, $0三个值推入栈中，接下来把%eax设置为SYS_exec然后执行int T_SYSCALL:这样做是告诉内核运行exec这个系统调用，如果运行正常的话，exec不会返回：它会运行名为$init的程序，$init是一个以空字符结尾的字符串，即/init(7721-7723)。如果exec失败并且返回了，initcode会不断调用一个不会返回的系统调用exit

系统调用exec的参数是$init\$argv。最后的0让这个手动创建的系统调用看起来就像普通的系统调用一样。

这样做是xv6努力避免为第一个进程的运行单独写一段代码，而是尽量使用通用于普通操作的代码。

对于exec的具体实现，概括地讲，它会用从文件系统中获取的/init的二进制代码代替initcode的代码，现在initcode已经执行完了，进程将要运行/init。/init(7810)会在需要的情况下创建一个新的控制台设备文件，然后把它作为描述符0，1，2打开。接下来它将不断循环，开启的控制台shell，处理没有父进程的僵尸进程，直到shell退出，然后再反复，系统就这样建立起来了。

在UNIX早期shell发明人道格拉斯·麦克罗伊发现，系统操作执行命令的时候，经常需要要将一个程序的输出交给另一个程序进行处理

但是这样的操作有些麻烦，于是出现了管道

这样前面写完后面读，这样就形成了通信。

实际上管道的设计也是遵循UNIX“一切皆文件”的设计原则的，它本质上就是一个文件，LINUX直接把管道实现成了一种文件系统，借助VFS（Virtual File System）给应用程序提供操作接口。

虽然pipeline实现形态上是文件，但是它本身并不占用磁盘或者其他外部存储的空间

在LINUX的实现上，它占用的是内存空间，所以LINUX上的管道就是一个操作方式

在fork之后，父进程和子进程都有了指向管道的文件描述符，子进程将管道的读端口拷贝在描述符0上，关闭p中的描述符，然后执行wc，当wc从标准输入读取时，它实际上是从管道读取的。父进程向管道的写端口写入然后关闭它的两个文件描述符。

如果数据没有准备好，那么对管道执行的read会一直等待，直到有数据了或者其他绑定在这个管道写端口的描述符都已经关闭了。

在后一种情况中，read会返回0，就像是一份文件读到了最后，读操作会一直阻塞直到不可能再有新数据到来，这就是为什么我们在执行wc之前要关闭子进程的写端口。

如果wc指向了一个管道的写端口，那么wc永远也看不到EOF了。
