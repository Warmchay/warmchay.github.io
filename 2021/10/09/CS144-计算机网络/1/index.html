Network Apps

关于应用层上的通讯为：

  • Bidirectional，reliable byte stream – 双向&可靠的字节流

  • network的抽象 – 只是在两个程序间有个pipe

  • 应用层的实例

    • World Wide Web（HTTP）
    • Skype
    • BitTorrent

网络通讯非常核心的概念： Internet 间是通过 bidirectional reliable byte stream 来互相通信的

3种不同的 network apps 为：

  • World Wide Web： 以HTTP为例[1]，Client和Server直接依赖HTTP相互传送

  • BitTorrent（peer to peer）[2]： 有一个Tracker记录所有client的track，当新的client进来时，需要下载先torrent，torrent是基于Http的，接着tracker会发送一个client list给这个new client，new client就可以和其他的client互相通讯了。

    BitTorrent最大的特点是赋予了dense graph connection between clients，和传统的server和client不同，clients之间可以动态的交换数据

  • Skype： Skype对于3种不同的通讯场景有不同的策略

    1. server-client，用于类似于Http的传输方式进行数据传输

    2. client 1 - (NAT-client 2): 此时client 2由于安全的因素在NAT（Net Address Translation 网络掩蔽、IP掩蔽）之后，需要额外引入rendezvous server来解决这个问题，client 1想要和client 2进行通讯，client 1 需要发信息给 rendezvous server。因为client 2 是和rendezvous server相连的，rendezvous server会告诉client2有client 1要和它进行通讯的消息，如果client 2 接受了这个call，就可以和client 1建立连接，因为client 1是没有NAT的，所以可以很容易地与外部建立连接，这样的通讯方式又被称为 reverse connection。

      The Rendezvous(集成) Server provides queued delivery of binary messages from one client to a second,

      Rendezvous Server 提供从一个客户端到第二个客户端的二进制消息的排队传送。

    3. (client 1 -NAT)-(NAT-client 2): 因为两个clients都有NAT，都不容易与Client建立连接，因此引入了relay server作为中转站，并且relay server已经提前与client 12 都建立了连接，可以在这种场景下完成通讯。

      Relay server:中继服务器，方便用户去send emails.

Appendixes

A.1 Http and Https

1.1 Http

网络协议： PC间网络通信间达成的某种“约定”或“规则”，根据这种“约定”，不同生产设备/操作系统的PC间可以实现通信。

HTTP协议： Hyper Text Transfer Protocol（超文本传输协议），从Web服务器传输HTML到本地浏览器的传送协议，它提供了一种发布和接受HTML页面的方法。

HTTP原理： HTTP是基于TCP/IP通信协议来传递数据的协议，传输数据类型为HTML文件、团图片文件和查询结果等。它一般用于B/S(Browser.Server,浏览器/服务器模式)架构，浏览器作为HTTP客户端通过URL向HTTP服务端即WEB服务器发送所有请求。以访问百度为例：

（1）Http特点

  1. 支持C/S架构：也是一种请求/响应模式的协议
  2. 简单快速： 客户向服务器请求服务时，只需要传送「请求方法」和路径即可，请求方法常用的是GET、HEAD、POST；
  3. 灵活： HTTP允许传输任意类型的数据对象，数据类型用Conten-Type标记；
  4. 无连接： 限制每次连接只处理一个请求，服务器处理完请求，并收到客户的应答后，再断开连接。但是这样不利于客户端与服务器保持会话连接，因此为了弥补这种不足，目前有用Cookie和Session来记录http状态；
  5. 无状态： 无状态指协议对于事务处理没有记忆，后续处理需要前面的信息，需要重新传递信息。

（2）URI和URL的区别

Http使用URI（Uniform Resource Identifiers，URI）来传输数据和建立连接。

  • **URI： ** Uniform Resource Identifiers,统一资源标识符
  • URL： Uniform Resource Location 统一资源定位符

URI来标识一个具体资源是什么，URL来定位资源是什么

（3）Http报文组成

1⃣️ 请求报文构成
  1. 请求行： 包括请求方法、URL、协议/版本
  2. 请求头（Request Header）
  3. 请求正文
2⃣️ 响应报文构成
  1. 状态行
  2. 响应头
  3. 响应正文

（3）Http 常见请求方法

  • GET： 请求指定的页面信息，并返回实体主体；

  • POST： 向指定资源提交数据进行处理请求（如上传文件）。数据被包含在了请求体中，POST请求可能会导致新资源的建立或已有资源的修改；

  • HEAD： 类似于GET请求，但是返回的响应中只有报头；

  • PUT： 从 Client 向 Server 传送的数据取代指定文档的内容

  • DELETE： 请求服务器删除指定的页面

1⃣️ Post请求
2⃣️ Get请求

Post 和 Get 区别

  • 都包含请求头/请求行和请求正文（body）
  • Get用于查询，不对服务器上内容起作用，Post用于提交，如将密码放于body中
  • Get 直接添加到URL后，可以直接在URL中看到内容，Post放于报文内部，用户无法直接看到
  • Get 提交的数据长度有限制，因为URL长度有限制；具体长度由浏览器决定。POST没有限制

（4）Http响应状态码

当浏览器向Web服务器发出请求时，网页所在服务器会返回一个包含HTTP状态码的信息头来响应浏览器的请求

状态码分类

类型 回应
信息型 1XX 服务器收到请求，需要请求者继续操作
成功型 2XX 请求成功收到，理解并处理
重定向 3XX 需要进一步操作来完成请求
客户端错误 4XX 存在语法错误 | 无法完成请求
服务端错误 5XX Server处理请求过程中发生了错误

HTTP 常见状态码

目前绝大多数网站采用的都为 Https 协议， Why？

Http 存在如下问题：

  1. 请求信息为明文传输，容易被窃听截取
  2. 数据完整性没有被校验，容易被篡改
  3. 不能验证对方身份，存在冒充危险

1.2 Https

Https: HyperText Transfer Protocol over Secure Socket Layer

为：Http+SSL/TLS，通过SSL证书来验证服务器身份，为浏览器和服务器之间的通信进行加密

SSL（Secure Socket Layer）： 安全套接字层，在TCP/IP协议和各种应用层协议之间，为数据通讯提供安全支持

TLS（Transport Layer Security): 传输层安全性协议，采用非对称加密对通信方做身份认证，目前Google、FaceBook在以这个协议来创建安全连线。

Https 数据传输流程：

KeyPoints：

  • Client 通过 URL 访问 Server 建立 SSL 连接
  • 客户端将网站的公钥加密为会话密钥，后传送给网站
  • 服务器通过会话密钥加密其与Client之间的通信

缺陷：

  1. 多次握手，导致页面加载时间延长了近50%
  2. 连接缓存没有Http高效，增加了数据开销和功耗
  3. 申请SSL需要money，功能越强大的证书费用会越高
  4. SSL涉及到的安全算法会消耗CPU资源，消耗服务器资源较大

Https 与 Http 的区别

  1. Http协议的数据传输为明文，不安全；Https用SSL/TLS协议对内容进行了加密处理
  2. Http和Https使用连接的方式也不同，Http默认端口为80，Https默认端口为443

A.2 BitTorrent

1. 相关概念

  • Tracker： 收集下载者信息的服务器，并把这个信息提供给其他的下载者，让下载者们集合起来一起传输数据；
  • 种子： 如果一个下载任务中的所有文件都被下载者A完整的download了，那么此时A成为一个种子，发布者本身发布的文件就是原始种子；
  • 做种
    1. 发布者提供了下载任务的全部内容
    2. 下载者下载完成后继续提供给其他人下载

2. 过程

文件发布者根据要发布的文件生成一个.torrent文件，即“种子”；

种子包含了Tracker信息和文件的信息，主要原理为把下载的文件虚拟分成大小相等的块，并且把每个块的索引信息和Hash验证码写入种子中，此时种子文件即为下载文件的“index”。

下载者要下载文件内容，需要先得到相应的种子文件，再使用BT客户端软件下载

BT客户端先解析种子文件得到Tracker地址，后连接到Tracker服务器，Tracker服务器回应下载者请求，提供下载者和其他下载者的IP，下载者再连接其他下载者，根据种子文件，双方交换彼此没有的数据。

因为这个过程不需要其他服务器参与，分散了单个线路上的数据流量，减轻了服务器的负担，BT下载的特点是：下载的人越多，提供的带宽也越多，下载速度就会越快。
